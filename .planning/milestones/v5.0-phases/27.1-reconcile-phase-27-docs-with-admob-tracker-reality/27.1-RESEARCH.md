# Phase 27.1: Reconcile Phase 27 docs with AdMob tracker reality - Research

**Researched:** 2026-04-07
**Domain:** iOS Privacy Manifest (PrivacyInfo.xcprivacy), App Store compliance documentation, Google AdMob SDK integration
**Confidence:** HIGH

## Summary

Phase 27.1 is a documentation reconciliation phase correcting a critical gap in Phase 27's compliance work: Phase 27 disclosed Spoonacular as a data processor but never disclosed Google AdMob as an active tracker, despite AdMob SDK runtime presence confirmed in `AdClient.swift` and `ConsentClient.swift`. The phase modifies five artifacts (PrivacyInfo.xcprivacy, POLICY-UPDATE.md, 27-01-SUMMARY.md, notes.txt, 27.1-VERIFICATION.md) to bring declared compliance into alignment with actual app behavior.

The research confirms that all decisions in CONTEXT.md are technically sound and align with current Apple Privacy Manifest schema (as of April 2026) and Google AdMob best practices. The existing codebase already has full AdMob runtime integration (GADMobileAds SDK initialization, ATT via AppTrackingTransparency, UMP SDK for GDPR consent), and the privacy manifest already declares `NSPrivacyTracking=true` with 5 Google tracking domains from Phase 20 work. Phase 27.1 adds the MISSING data type declarations (Device ID, Advertising Data, second Coarse Location, Other Diagnostic Data) and updates human-facing documentation to match.

**Primary recommendation:** Execute Phase 27.1 as a single-plan atomic operation. All five artifact edits are tightly coupled (the ASC checklist in 27-01-SUMMARY.md references the manifest entries, the POLICY-UPDATE.md mirrors the manifest structure, notes.txt references both). Splitting into multiple plans risks partial-state documentation drift.

### Pattern 1: Duplicate Data Type with Different Purpose

**What:** Apple's privacy manifest schema allows multiple dictionaries in `NSPrivacyCollectedDataTypes` array with the same `NSPrivacyCollectedDataType` value but different `NSPrivacyCollectedDataTypeTracking` / `NSPrivacyCollectedDataTypePurposes` values.

**When to use:** When a single data type (e.g., Coarse Location) is collected for multiple purposes by different subsystems (Mapbox for app functionality, AdMob for ad targeting). Collapsing into one entry loses granularity; using separate entries maps 1:1 to the App Store Connect privacy label rows.

**Example (Phase 27.1 Coarse Location):**

```xml
<!-- Existing entry #2 from Phase 27 (Mapbox) — DO NOT MODIFY -->
<dict>
	<key>NSPrivacyCollectedDataType</key>
	<string>NSPrivacyCollectedDataTypeCoarseLocation</string>
	<key>NSPrivacyCollectedDataTypeLinked</key>
	<false/>
	<key>NSPrivacyCollectedDataTypeTracking</key>
	<false/>
	<key>NSPrivacyCollectedDataTypePurposes</key>
	<array>
		<string>NSPrivacyCollectedDataTypePurposeAppFunctionality</string>
	</array>
</dict>

<!-- New entry #11 for AdMob — ADD after Search History -->
<dict>
	<key>NSPrivacyCollectedDataType</key>
	<string>NSPrivacyCollectedDataTypeCoarseLocation</string>
	<key>NSPrivacyCollectedDataTypeLinked</key>
	<false/>
	<key>NSPrivacyCollectedDataTypeTracking</key>
	<true/>  <!-- Different from Mapbox entry -->
	<key>NSPrivacyCollectedDataTypePurposes</key>
	<array>
		<string>NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising</string>
	</array>
</dict>
```

**Rationale:** Apple's App Store Connect privacy form allows creating two "Coarse Location" rows — one under "Data Not Linked to You" (Mapbox), one under "Data Used to Track You" (AdMob). The manifest structure must support this 1:1 mapping.

### Pattern 2: NSPrivacyTracking Unconditional Declaration

**What:** For data types with `Tracking = true`, the privacy manifest declares the SDK's POTENTIAL to collect that data, not the actual runtime behavior conditioned on user consent.

**When to use:** Always, per Apple's static manifest model. Even though Kindred gates AdMob personalization with ATT (`publisherPrivacyPersonalizationState = .disabled` when ATT denied), the manifest must declare `Tracking = true` for Device ID and Advertising Data because the SDK CAN collect them when authorized.

**Example:**

```xml
<!-- Device ID (IDFA) — declared even if user denies ATT -->
<dict>
	<key>NSPrivacyCollectedDataType</key>
	<string>NSPrivacyCollectedDataTypeDeviceID</string>
	<key>NSPrivacyCollectedDataTypeTracking</key>
	<true/>  <!-- Unconditional, regardless of ATT runtime state -->
	<!-- ... -->
</dict>
```

**Comment block style (from CONTEXT.md):**

```xml
<!-- 9-12. Google AdMob Tracking Data (Free-Tier Banner Ads) -->
<!-- AdMob SDK (Google Mobile Ads 11.2.0+) displays banner ads to free-tier users. Pro subscribers see no ads (AdClient.shouldShowAds checks subscription state). Personalized ads require ATT authorization (AppTrackingTransparency); denying ATT sets publisherPrivacyPersonalizationState=.disabled. EU users consent via UMP SDK (Google User Messaging Platform). These declarations cover AdMob's POTENTIAL data collection regardless of user consent state (Apple's static manifest model). -->
```

### Anti-Patterns to Avoid

- **Modifying the existing Mapbox Coarse Location entry:** Phase 27.1 adds a SECOND Coarse Location dict. Do NOT change the existing entry's `Tracking` flag or `Purposes` array — that would break the Mapbox declaration from Phase 5.
- **Adding api.spoonacular.com to NSPrivacyTrackingDomains:** Spoonacular is a data processor (like a database), not a tracking partner. This was explicitly documented in Phase 27-01 and must not be contradicted in Phase 27.1.
- **Adding Required Reason API entries for Google SDKs:** Google Mobile Ads SDK 11+ and UMP SDK ship their own PrivacyInfo.xcprivacy bundles. Xcode aggregates these automatically. Adding redundant entries to the app's manifest risks conflicts or validation errors.
- **Speculative tracking domain additions:** Only add domains to NSPrivacyTrackingDomains if Google's official docs OR a runtime network trace confirms iOS egress to them. Do NOT add domains based on "might be used" reasoning.
- **Collapsing multiple data type purposes into one manifest entry:** Apple's App Store Connect form allows separate rows for the same data type with different purposes/tracking flags. The manifest should match this granularity (1 manifest dict = 1 ASC form row).

## Common Pitfalls

### Pitfall 1: Conflating "Data Processor" vs "Tracker" Semantics

**What goes wrong:** Planner or executor adds Spoonacular to NSPrivacyTrackingDomains OR marks Search History with `Tracking=true` because "it sends data to a third party."

**Why it happens:** Apple's distinction is subtle. "Data processor" (Spoonacular — provides recipe data via API) vs "tracker" (AdMob — joins data across apps/websites for ad targeting) maps to different manifest semantics. Phase 27 correctly excluded Spoonacular from tracking; Phase 27.1 must NOT contradict this.

**How to avoid:** CONTEXT.md explicitly locks: "Do NOT add api.spoonacular.com to NSPrivacyTrackingDomains." The planner must enforce this constraint in the plan's verification criteria. The executor must NOT modify NSPrivacyTrackingDomains unless conservative additions (new Google domains) are found.

**Warning signs:**
- Plan includes editing the existing NSPrivacyTrackingDomains array to add non-Google domains
- Plan marks Search History data type with `Tracking=true` (should be `false` per Phase 27-01)
- Verification report checks for "Spoonacular NOT in tracking domains" (correct) but doesn't check "AdMob entries ARE in tracking section" (missing the gap)

### Pitfall 2: Editing Phase 27 Artifacts Instead of Creating Phase 27.1 Artifacts

**What goes wrong:** Planner instructs executor to edit `.planning/phases/27-app-store-compliance/27-VERIFICATION.md` to "fix" deviation #3 or line 113, rewriting history.

**Why it happens:** Natural instinct to "correct the error at the source." But Phase 27 is CLOSED and its artifacts are the historical record of what was known/decided at that time. Phase 27.1 is the correction phase.

**How to avoid:** CONTEXT.md explicitly locks: "Write a new `27.1-VERIFICATION.md` in the Phase 27.1 directory (NOT in the Phase 27 directory)." The planner must create a new file, not edit the old one. The new file cross-references the old one with explicit "Phase 27 claim → Phase 27.1 correction" structure (Pattern 3 above).

**Warning signs:**
- Plan modifies `.planning/phases/27-app-store-compliance/27-VERIFICATION.md` instead of creating `.planning/phases/27.1-.../27.1-VERIFICATION.md`
- Verification report doesn't quote the original Phase 27 claims before stating corrections
- Git history shows edits to Phase 27 files committed under Phase 27.1 plan execution

### Pitfall 3: NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising Constant Typo

**What goes wrong:** Executor types `NSPrivacyCollectedDataTypePurposeAdvertising` (missing "ThirdParty" prefix) in the new manifest entries, causing App Store Connect label mismatch or validation warnings.

**Why it happens:** Apple's privacy purpose constants have verbose names. The older `NSPrivacyCollectedDataTypePurposeAdvertising` constant exists in some docs but `NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising` is the correct one for third-party ad networks like AdMob (per CONTEXT.md "preferred per Apple's recent docs").

**How to avoid:** CONTEXT.md gives Claude's Discretion to "pick whatever the existing PrivacyInfo schema validates against." Research found that `NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising` is the correct constant as of 2025-2026 docs. Planner must specify the EXACT string in the plan. Verification must grep for this exact string.

**Warning signs:**
- `plutil -p` output shows `NSPrivacyCollectedDataTypePurposeAdvertising` (missing "ThirdParty") for Device ID or Advertising Data entries
- App Store Connect privacy form import fails with "unrecognized purpose" error after Phase 28 upload
- WebSearch found "NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising" references in 2025-2026 contexts, confirming it's current

### Pitfall 4: ASC Checklist Wording Creates False Impression

**What goes wrong:** Step 6 rewrite says "The 'Data Used to Track You' section is EMPTY except for AdMob," which a user might interpret as "delete all existing entries and add only AdMob."

**Why it happens:** Ambiguous phrasing in compliance checklists. Phase 27's step 6 said "section is EMPTY" (implying nothing should be there). Phase 27.1 corrects this but must be crystal-clear that AdMob entries are ADDITIONS, not replacements.

**How to avoid:** CONTEXT.md specifies the exact wording: "Confirm Spoonacular is NOT in 'Data Used to Track You' (Spoonacular is a data processor, not a tracker). The section MUST contain AdMob-related entries — see steps 9-11 below." This avoids negative framing ("is EMPTY") and uses positive framing ("MUST contain"). Planner must preserve this wording verbatim.

**Warning signs:**
- Step 6 uses "EMPTY except for" phrasing (ambiguous)
- Steps 9-11 don't explicitly say "ADD" before each entry (user might think they're replacing)
- No banner at top of checklist flagging Phase 27.1 updates (user might miss the new steps)

### Pitfall 5: Tracking Domain Audit Adds Speculative Domains

**What goes wrong:** Planner researches Google AdMob domains and finds developer forum posts mentioning `fundingchoicesmessages.google.com`, `imasdk.googleapis.com`, etc., then adds all of them to NSPrivacyTrackingDomains without verification.

**Why it happens:** CONTEXT.md mentions these domains as "likely additions to investigate," which could be misread as "add these." But it also says "Conservative addition only — add a domain only if Google's published docs OR a runtime trace confirms iOS network egress to it."

**How to avoid:** Research found that Google does NOT publish an explicit NSPrivacyTrackingDomains list in AdMob iOS docs. The existing 5 domains from Phase 20 are already conservative. Unless the planner finds a Google developer guide that says "add X to NSPrivacyTrackingDomains," keep the existing 5. A runtime trace would require Proxyman/Charles on a physical device running the app, which is out of scope for this documentation phase.

**Warning signs:**
- Plan adds domains without citing a Google official docs URL
- Plan cites a Stack Overflow post or developer forum comment as source
- Verification grep checks for >5 domains without explaining which Google doc confirmed each addition

## State of the Art

### Privacy Manifest Schema (as of April 2026)

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| NSPrivacyCollectedDataTypePurposeAdvertising (first-party ads) | NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising (ad networks) | iOS 14 → iOS 15 transition (2021-2022) | Third-party ad SDKs like AdMob must use "ThirdPartyAdvertising" purpose. Using the older "Advertising" purpose may trigger App Store Connect warnings or mismatch with privacy labels. |
| Single Coarse Location entry with multiple purposes in array | Duplicate Coarse Location entries with separate purposes/tracking flags | No schema change, but best practice evolved (2023-2024 App Store rejections for granularity) | Apple's App Store Connect privacy form allows separate rows for same data type. Manifest should match this 1:1 to avoid "missing data type" rejections. |
| Manual NSPrivacyTrackingDomains population from network traces | Third-party SDKs ship bundled PrivacyInfo.xcprivacy (Xcode auto-aggregates) | iOS 17 / Xcode 15 (2023) | Apps using AdMob SDK 11.2.0+ get Google's tracking domain list automatically. App-level manifest only needs first-party tracking domains. Phase 27.1 audit verifies existing 5 domains are sufficient; Google's bundled manifest covers additional SDK-specific domains. |

**Deprecated/outdated:**
- **Single-purpose Coarse Location entry combining Mapbox + AdMob:** Pre-2024 apps often collapsed both into one manifest dict with `Purposes = [AppFunctionality, ThirdPartyAdvertising]`. Apple's 2024 enforcement tightened — reviewers now flag this as under-granular. Use separate dicts.
- **NSPrivacyAccessedAPITypes for third-party SDK APIs:** Pre-Xcode 15, apps declared Required Reason APIs for ALL code (first-party + dependencies). Xcode 15+ aggregates SDK-bundled manifests automatically. Declaring Google SDK APIs in the app manifest is now redundant.
- **Listing all possible Google domains from developer forums:** Early privacy manifest adopters (2023-2024) added 10+ Google domains from unverified sources. Google's official stance (2025-2026): SDK bundles its own manifest; app should only list domains IT directly connects to. Phase 20's 5 domains are conservative and sufficient.
