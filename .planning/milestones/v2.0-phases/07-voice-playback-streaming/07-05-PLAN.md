  <name>Task 1: Fix AudioPlayerManager.play() to wait for readyToPlay and add diagnostic error reporting</name>
  <files>Kindred/Packages/VoicePlaybackFeature/Sources/AudioPlayer/AudioPlayerManager.swift</files>
  <action>
    Fix the AudioPlayerManager so that AVPlayer properly loads and plays audio:

    **Root cause fix — use waitForReadyToPlay() in play():**

    The `waitForReadyToPlay(playerItem:)` method already exists in AudioPlayerManager (lines 213-253) but is NEVER called from `play()`. The play() method creates an AVPlayerItem, assigns it to AVPlayer, and immediately calls `player.play()` without waiting for the item to reach `.readyToPlay` status. This causes "Cannot Open" on items that haven't loaded yet.

    Modify `play(url:)` to:
    1. Create AVPlayerItem with the URL
    2. Set audioTimePitchAlgorithm to `.spectral`
    3. Create AVPlayer with the item
    4. Set automaticallyWaitsToMinimizeStalling = true, volume = 1.0
    5. **Call `try await waitForReadyToPlay(playerItem: playerItem)`** — this waits for the item's status to become .readyToPlay (with 15s timeout)
    6. ONLY THEN call `newPlayer.play()`
    7. If waitForReadyToPlay throws, throw a descriptive error including the URL and the AVPlayerItem.error

    **Enhanced error diagnostics:**

    After the `waitForReadyToPlay` call, if it fails, capture and throw an error that includes:
    - The attempted URL (url.absoluteString)
    - The AVPlayerItem.status (rawValue)
    - The AVPlayerItem.error?.localizedDescription if available
    - The AVAudioSession.sharedInstance().category and isOtherAudioPlaying status

    This ensures that if audio still fails after the fix, the error message in the UI will contain actionable diagnostic info.

    **Add audio session verification in play():**

    Before creating the AVPlayerItem, verify the audio session is active:
    ```swift
    let session = AVAudioSession.sharedInstance()
    logger.info("Audio session category: \(session.category.rawValue), mode: \(session.mode.rawValue), isOtherAudioPlaying: \(session.isOtherAudioPlaying)")

    // Re-activate session if needed (handles silent mode, etc.)
    if session.category != .playback {
        try session.setCategory(.playback, mode: .spokenAudio)
    }
    try session.setActive(true)
    ```

    **Fix statusStream() to check for AVPlayerItem errors:**

    In the `.paused` case of the KVO observer, also check if `player.currentItem?.status == .failed`:
    ```swift
    case .paused:
        if let error = player.currentItem?.error {
            .error("Cannot Open: \(error.localizedDescription)")
        } else if player.currentItem?.isPlaybackLikelyToKeepUp == false {
            .buffering
        } else {
            .paused
        }
    ```

    Also add observation for AVPlayerItem.status changes (not just timeControlStatus) to catch `.failed` status:
    ```swift
    // Observe item status for error detection
    let itemObservation = player.currentItem?.observe(\.status, options: [.new]) { item, _ in
        if item.status == .failed, let error = item.error {
            continuation.yield(.error("Failed: \(error.localizedDescription)"))
        }
    }
    ```

    **Remove or rename TestAudioGenerator concerns:**

    Do NOT delete TestAudioGenerator — it's used by the reducer for test audio. But ensure it generates a valid WAV file. The current implementation looks correct (AVAudioFile + PCM buffer), so the issue is likely the play() lifecycle, not the file generation.

    **Logging improvements:**

    - When play(url:) is called — log the URL
    - When waitForReadyToPlay succeeds — log "ready to play"
    - When player.play() is called — log "play() called"
    - When any error occurs — log the full error chain

<task type="auto">
  <name>Task 2: Fix VoicePlaybackReducer — remove premature status, enable cache, remove destructive clearCache</name>
  <files>Kindred/Packages/VoicePlaybackFeature/Sources/Player/VoicePlaybackReducer.swift</files>
  <action>
    Fix three issues in VoicePlaybackReducer that prevent proper playback and caching:

    **Fix 1: Remove premature `.statusChanged(.playing)` (line ~332)**

    In the `.narrationReady` case, after calling `audioPlayer.play(url)`, there is a manual:
    ```swift
    await send(.statusChanged(.playing))
    ```
    This MUST be REMOVED. The actual playback status should come from the `statusStream()` observation (the task group below). The premature status causes the reducer to believe audio is playing when it may still be loading or may have failed.

    After removing the manual status send, the status flow will be:
    1. `.narrationReady` sets status to `.loading`
    2. `audioPlayer.play(url)` starts AVPlayer (now waits for readyToPlay per Task 1)
    3. `statusStream()` emits `.buffering` while loading
    4. `statusStream()` emits `.playing` when audio actually plays
    5. If error: `statusStream()` emits `.error(message)` which the reducer already handles

    **Fix 2: Enable cache operations (line ~271)**

    Change `if false, let cachedURL = ...` to `if let cachedURL = ...`:
    ```swift
    // BEFORE (disabled):
    if false, let cachedURL = await voiceCache.getCachedAudio(voiceId, recipeId) {

    // AFTER (enabled):
    if let cachedURL = await voiceCache.getCachedAudio(voiceId, recipeId) {
    ```

    This re-enables the cache-first strategy: check if audio is already cached before streaming from remote.

    **Fix 3: Remove destructive `clearCache()` call (line ~268)**

    Remove this line entirely:
    ```swift
    try? await voiceCache.clearCache()
    ```
    This was added during debugging and clears the ENTIRE cache on every voice selection. This is destructive and defeats the purpose of caching. Remove it.

    **Fix 4: Ensure stream observations use .cancellable with correct IDs**

    Currently the entire `.run` effect in `.narrationReady` is cancelled with `.cancellable(id: CancelID.timeObserver)`. This cancels ALL three stream observations (time, status, duration) when only the time observer is cancelled.

    Split the task group into separate `.run` effects with individual cancel IDs:
    ```swift
    return .merge(
        .run { send in
            guard let url = URL(string: metadata.audioURL) else {
                await send(.narrationFailed("Invalid audio URL"))
                return
            }
            do {
                try await audioPlayer.play(url)
            } catch {
                await send(.narrationFailed("\(url.absoluteString) — \(error.localizedDescription)"))
            }
        },
        .run { send in
            for await time in await audioPlayer.currentTimeStream() {
                await send(.timeUpdated(time))
            }
        }
        .cancellable(id: CancelID.timeObserver),
        .run { send in
            for await status in await audioPlayer.statusStream() {
                await send(.statusChanged(status))
            }
        }
        .cancellable(id: CancelID.statusObserver),
        .run { send in
            for await duration in await audioPlayer.durationStream() {
                await send(.durationUpdated(duration))
            }
        }
        .cancellable(id: CancelID.durationObserver)
    )
    ```

    This ensures each stream is independently cancellable, and the play() call is separate from the observation effects.
  </action>
