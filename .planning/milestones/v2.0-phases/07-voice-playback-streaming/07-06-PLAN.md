    **Scenario B: "Cannot Open" still occurs with diagnostic info**
    Read the enhanced error message. Common residual causes:

    1. **File URL scheme issue:** TestAudioGenerator returns a file:// URL as string via `.absoluteString`. When the reducer does `URL(string: metadata.audioURL)` it reconstructs the URL. If the path contains spaces or special characters, this can fail. Fix: In the selectVoice case, pass the URL object directly instead of converting to string and back. Or ensure NarrationMetadata stores a properly encoded URL string.

    2. **AVAudioSession not active:** If diagnostic shows `category != .playback`, the AudioSessionConfigurator.configure() may have failed silently at app launch. Fix: Add `do/try/catch` with explicit error logging in AudioSessionConfigurator.configure(), and re-activate in play().

    3. **Sandboxing / file access:** Local file:// URLs in the caches directory should be accessible. If not, check that TestAudioGenerator creates the file successfully (log file existence check).

    4. **Actor isolation issue:** AudioPlayerManager is an actor. The `currentTimeStream()` and `statusStream()` methods access `self.player` inside an AsyncStream closure, but those closures run on the caller's context, not the actor. This means `self.player` may be nil when the closures execute if there's a timing gap. Fix: Capture the player reference in a local variable BEFORE creating the AsyncStream:
    ```swift
    public func currentTimeStream() -> AsyncStream<TimeInterval> {
        guard let player = self.player else {
            return AsyncStream { $0.finish() }
        }
        let capturedPlayer = player  // Capture outside closure
        return AsyncStream { continuation in
            let token = capturedPlayer.addPeriodicTimeObserver(...)
            ...
        }
    }
    ```

    5. **WAV format not supported by AVPlayer:** AVPlayer may not support raw PCM WAV format. If this is the case, modify TestAudioGenerator to output AAC/M4A format instead, or use `AVAudioEngine` to convert. Alternatively, use a known-good remote test URL like a public MP3.

    **Scenario C: play() throws timeout error (15 seconds)**
    The item never reaches .readyToPlay. This suggests:
    - The URL is unreachable (file doesn't exist, or remote URL is invalid)
    - The audio format is not supported by AVPlayer
    Fix: Verify file existence before creating AVPlayerItem. Add explicit format logging.

    **Regardless of scenario, ensure:**
    - All `var` → `let` where mutation isn't needed (clean up unused `var currentPlayback` warnings in reducer)
    - Logger output is clean and informative
    - The test audio URL path is logged so it can be verified manually
