    2. Create `backend/src/images/r2-storage.service.ts`:
       - Inject ConfigService for R2_ACCESS_KEY_ID, R2_SECRET_ACCESS_KEY, CLOUDFLARE_ACCOUNT_ID, R2_BUCKET_NAME, R2_PUBLIC_URL
       - Initialize S3Client with R2 endpoint: `https://{CLOUDFLARE_ACCOUNT_ID}.r2.cloudflarestorage.com`, region 'auto'
       - `uploadImage(imageBuffer: Buffer, key: string, contentType: string = 'image/jpeg'): Promise<string>`:
         - Use PutObjectCommand to upload to R2 bucket
         - Return public URL: `{R2_PUBLIC_URL}/{key}`
       - `deleteImage(key: string): Promise<void>`:
         - Use DeleteObjectCommand to remove from R2 (for re-generation or cleanup)
       - Handle connection errors gracefully, log and rethrow for caller to handle

    3. Create `backend/src/images/images.service.ts`:
       - Inject R2StorageService, PrismaService, ConfigService, Logger
       - `generateAndStoreImage(recipeId: string, recipeName: string, ingredients: string[]): Promise<string | null>`:
         a. Update recipe imageStatus to GENERATING
         b. Build Imagen 4 Fast prompt: "Professional flat lay top-down food photograph of {recipeName}. Key ingredients ({top 5 ingredients joined}) arranged artistically around the finished dish. Clean white marble surface, natural soft lighting, warm tones, Instagram-worthy editorial food photography style. No text, no watermarks, no hands."
         c. Call Imagen 4 Fast API via Vertex AI (use @google-cloud/aiplatform):
            - Project: GOOGLE_CLOUD_PROJECT
            - Location: 'us-central1'
            - Model: 'imagen-4-fast' (or 'imagegeneration@006' if that is the endpoint name)
            - 1 image, 1:1 aspect ratio
         d. On success: upload buffer to R2 with key `recipes/{recipeId}/hero.jpg`
         e. Update recipe: imageUrl = R2 public URL, imageStatus = COMPLETED
         f. On failure: update recipe imageStatus = FAILED, log error with recipeId
         g. Return imageUrl or null
       - `retryFailedImages(limit: number = 10): Promise<number>`:
         - Find recipes with imageStatus FAILED, retry generation for up to limit recipes
         - Return count of successfully generated images

    4. Create `backend/src/images/images.module.ts`:
       - Import PrismaModule, ConfigModule
       - Provide ImagesService, R2StorageService
       - Export ImagesService (scraping module needs it)

  <action>
    1. Create `backend/src/images/image-generation.processor.ts`:
       - Simple in-memory queue processor (no BullMQ/Redis needed for single-instance MVP):
         - Maintain a queue (array) of pending image generation jobs: { recipeId, recipeName, ingredients }
         - `enqueue(job: ImageJob): void` -- add job to queue
         - `processQueue(): Promise<void>` -- process jobs sequentially with configurable concurrency (default 3 parallel)
         - Use setInterval or @nestjs/schedule @Interval(5000) to check queue every 5 seconds
         - For each job: call ImagesService.generateAndStoreImage
         - Rate limit: max 10 images per minute to stay within Imagen API limits
         - Log processing status: "Processing image {n}/{total} for recipe {name}"
       - This avoids Redis/BullMQ complexity for MVP. Can upgrade to BullMQ when scaling to multiple instances (noted in RESEARCH.md open questions).

    2. Update `backend/src/scraping/scraping.service.ts`:
       - Inject ImageGenerationProcessor
       - After successfully creating a recipe in `scrapeForCity()`, enqueue an image generation job:

         ```
         this.imageProcessor.enqueue({
           recipeId: createdRecipe.id,
           recipeName: createdRecipe.name,
           ingredients: parsedRecipe.ingredients.map(i => i.name),
         });
         ```
       - This decouples scraping from image generation per RESEARCH.md anti-pattern guidance ("Synchronous Imagen calls in resolver blocks GraphQL response")
