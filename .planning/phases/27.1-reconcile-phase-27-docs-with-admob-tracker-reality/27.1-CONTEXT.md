# Phase 27.1: Reconcile Phase 27 docs with AdMob tracker reality - Context

**Gathered:** 2026-04-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 27 was scoped narrowly to **Spoonacular as a data processor** and never disclosed that **Google AdMob is an active tracker** in the Kindred codebase. Phase 27.1 reconciles this gap across the four compliance artifacts that Phase 27 produced or modified, so the App Store submission posture matches the actual runtime behavior of the app.

**The reality (verified by code scout 2026-04-07):**
- `Kindred/Packages/MonetizationFeature/Sources/Ads/AdClient.swift` calls `GoogleMobileAds.GADMobileAds.sharedInstance().start` and `configurePersonalization` per consent state
- `Kindred/Packages/MonetizationFeature/Sources/Consent/ConsentClient.swift` runs ATT (`ATTrackingManager.requestTrackingAuthorization`) + UMP (`UMPConsentForm.loadAndPresentIfRequired`) flow
- `Kindred/Sources/PrivacyInfo.xcprivacy` declares `NSPrivacyTracking=true` and 5 Google ad/analytics tracking domains (Phase 20 work)
- Production AdMob unit IDs are present in `Kindred/Config/Release.xcconfig` (commit `ff128c4`)
- AdMob test banner was visually observed in `tr/05-recipe-detail.png` during Phase 27 verification (deviation #3)

**The gaps in Phase 27 docs:**
1. `PrivacyInfo.xcprivacy` has tracking domains but **no ad-related collected data type entries**
2. `POLICY-UPDATE.md` lists only Spoonacular + ElevenLabs — no AdMob disclosure at all
3. `27-01-SUMMARY.md` ASC checklist (step 6) tells the user to keep "Data Used to Track You" EMPTY, which is a rejection risk because AdMob does track
4. `27-VERIFICATION.md` deviation #3 dismisses the AdMob banner observation as "informational, not a compliance issue" — that conclusion was wrong in light of the missing disclosures
5. `Kindred/fastlane/metadata/review_information/notes.txt` only mentions the Spoonacular backend proxy

**In scope for Phase 27.1:**
- `Kindred/Sources/PrivacyInfo.xcprivacy` — add AdMob-related data type declarations + audit `NSPrivacyTrackingDomains`
- `.planning/phases/27-app-store-compliance/POLICY-UPDATE.md` — add Google AdMob third-party processor section
- `.planning/phases/27-app-store-compliance/27-01-SUMMARY.md` — fix the ASC checklist (step 6 + add explicit AdMob "Data Used to Track You" steps)
- `.planning/phases/27-app-store-compliance/27-VERIFICATION.md` — correct deviation #3 OR write new `27.1-VERIFICATION.md` cross-referencing the corrections (Claude's discretion)
- `Kindred/fastlane/metadata/review_information/notes.txt` — append AdMob disclosure addendum

**Not in scope:**
- AdMob runtime behavior changes (no AdClient/ConsentReducer edits)
- New ad placements, removal of ad placements, or ad-format additions
- ATT pre-prompt copy review or UMP form customization
- Firebase Analytics user-property wiring (the unresolved gap noted in 27-01-SUMMARY)
- App Store description (`fastlane/metadata/{en-US,tr}/description.txt`) edits — defer to Phase 28 release prep
- Hosted privacy policy publishing — that's a manual user step before Phase 28
- A new Phase 27 plans (27-05, 27-06 etc.) — Phase 27 stays closed; 27.1 produces its own plan(s)

<decisions>
## Implementation Decisions

### PrivacyInfo.xcprivacy data type additions

**Add four new entries to `NSPrivacyCollectedDataTypes`:**

1. **Device ID (IDFA)** — `NSPrivacyCollectedDataTypeDeviceID`
   - `Linked = false`
   - `Tracking = true`
   - Purpose: `NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising`
   - Rationale: AdMob requests IDFA via ATT when authorized; this is the canonical AdMob declaration.

2. **Advertising Data** — `NSPrivacyCollectedDataTypeAdvertisingData`
   - `Linked = false`
   - `Tracking = true`
   - Purpose: `NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising`
   - Rationale: Apple's "Information about advertisements the user has seen" — covers ad impressions/clicks AdMob reports back to Google.

3. **Coarse Location (second entry, AdMob purpose)** — `NSPrivacyCollectedDataTypeCoarseLocation`
   - **Add a SECOND CoarseLocation dict** — do NOT modify the existing Mapbox entry.
   - `Linked = false`
   - `Tracking = true`
   - Purpose: `NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising`
   - Rationale: AdMob may use coarse location for ad targeting depending on personalization config. Keeping the Mapbox entry untouched preserves the App Functionality declaration; the duplicate-with-different-purposes pattern is allowed by Apple's manifest schema and is more accurate than collapsing both into one row.

4. **Other Diagnostic Data** — `NSPrivacyCollectedDataTypeOtherDiagnosticData`
   - `Linked = false`
   - `Tracking = false`
   - Purpose: `NSPrivacyCollectedDataTypePurposeAnalytics`
   - Rationale: AdMob SDK telemetry / performance metrics. Lower-risk but still under-declared today.

**ATT semantics — declare unconditionally:**
- All `Tracking = true` entries above are declared regardless of the user's runtime ATT choice. Apple's static manifest model requires declaring what the SDK CAN collect under any user choice, not what is collected per session. This aligns with how AdMob-using apps publish their manifests.
- Document the ATT gate (and the Pro-tier ad suppression in `AdClient.shouldShowAds`) in the manifest comment block, not in the manifest data itself.

### NSPrivacyTrackingDomains audit

- Cross-reference Google's official AdMob domain list against the existing 5 domains (`googleads.g.doubleclick.net`, `pagead2.googlesyndication.com`, `www.googleadservices.com`, `www.google-analytics.com`, `app-measurement.com`).
- Likely additions to investigate during planning/research: `imasdk.googleapis.com`, `tpc.googlesyndication.com`, `fundingchoicesmessages.google.com` (UMP), `csi.gstatic.com`.
- **Conservative addition only** — add a domain only if Google's published docs OR a runtime trace confirms iOS network egress to it. No speculative additions.

### Required Reason API audit (NSPrivacyAccessedAPITypes)

- **No action.** Google Mobile Ads SDK 11+ and UMP SDK both ship their own `PrivacyInfo.xcprivacy` bundles inside their xcframeworks. Apple aggregates these automatically. Adding entries for Google SDK APIs to our manifest would be redundant and may conflict.
- The first-party audit from Phase 27 (UserDefaults `CA92.1`, FileTimestamp `C617.1`) stays unchanged.

### Manifest comment style

- **Single anchor block comment + per-entry one-liners.** One block comment above the AdMob entries explains the SDK + ATT gate + Pro suppression. Each new data type dict has a 1-line comment with its specific rationale. Matches the existing Spoonacular comment style at `PrivacyInfo.xcprivacy:120-121`.

### POLICY-UPDATE.md AdMob section

- **Add a third subsection** to the existing `## Third-Party Data Processors` block, alongside Spoonacular and ElevenLabs. Title: `### Google AdMob (Advertising Network)`.
- Mirror the Spoonacular subsection structure: **Purpose, Data sent, User control / consent, Data retention, Privacy policy link.**
- Required content:
  - Plain-language statement that Kindred shows banner ads on the free tier and that Pro subscribers see no ads.
  - Disclosure that ATT (App Tracking Transparency) is requested before any advertising identifier is shared, and that denying ATT disables personalized advertising via `publisherPrivacyPersonalizationState = .disabled`.
  - Mention that EU users see the UMP consent form for GDPR purposes.
  - Link to https://policies.google.com/technologies/ads
  - Link to https://support.google.com/admob/answer/6128543 (AdMob privacy)
- Update the `## Summary of Changes in This Update` section at the top: bump `Version` from `2.0` to `2.1` and add a new bullet "Added Google AdMob disclosure (advertising network, IDFA via ATT, UMP for EU)."
- Update the `## Data We Collect → Other Data Types` bullet list to include Device ID, Advertising Data, and (second) Coarse Location entries marked Tracking=true.

<specifics>
## Specific Ideas

- "AdMob is a tracker, Spoonacular is a processor" is the anchor framing for the whole reconciliation. Phase 27 collapsed both into the wrong column; Phase 27.1 fixes the column headers.
- The four new manifest data types (Device ID, Advertising Data, second Coarse Location, Other Diagnostic Data) match the canonical AdMob declaration pattern published by Google in their iOS SDK docs.
- The "second Coarse Location entry" pattern is the key non-obvious decision — the planner/executor must NOT modify the existing Mapbox entry, only ADD a duplicate with different purposes/tracking.
- The ASC checklist correction is the highest-leverage human-facing fix — a user following the current checklist would submit the app with the tracking section empty and trigger an Apple review back-and-forth.
- The 27.1-VERIFICATION.md cross-reference pattern lets future readers find the corrected facts without rewriting Phase 27's history.
- Documentation tone in POLICY-UPDATE.md should stay consistent with existing Spoonacular/ElevenLabs prose: plain English, not legalese, mirroring the Kindred voice.
