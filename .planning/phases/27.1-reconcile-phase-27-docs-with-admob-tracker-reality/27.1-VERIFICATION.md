## Cross-Reference: Phase 27 Corrections

Phase 27.1 corrects two specific claims in `.planning/phases/27-app-store-compliance/27-VERIFICATION.md`. Both corrections are documented here, NOT by editing the Phase 27 file in place. The Phase 27 audit trail is preserved as a historical record.

### Correction 1 — Line 114 "Data Used to Track You" Expectation

**Phase 27 claim** (`27-VERIFICATION.md` line 114, under the "ASC Privacy Form Sync" expected results):

> `- "Data Used to Track You" section remains EMPTY (do not add Spoonacular)`

**Why it was wrong:** The claim is half-correct. Spoonacular should NOT be added to tracking (it is a data processor, not a tracker). But the section should NOT be empty — Google AdMob is an active tracker in the app (banner ads on free tier, IDFA via ATT, EU consent via UMP) and Phase 27 never disclosed it.

**Phase 27.1 correction:** The "Data Used to Track You" section MUST contain three AdMob entries — Device ID, Advertising Data, and a second Coarse Location entry — all marked Third-Party Advertising. Spoonacular correctly stays out of the tracking section. The corrected expectation is captured in `27-01-SUMMARY.md` step 6 (reworded) and steps 9-11 (added by this phase).

**Authoritative reference going forward:** Use `27-01-SUMMARY.md` (as updated by Phase 27.1) for the ASC checklist, not `27-VERIFICATION.md` line 114.

### Correction 2 — Deviation #3 "AdMob Test Banner Visible in tr/05-recipe-detail.png"

**Phase 27 claim** (`27-VERIFICATION.md` lines 196-204):

> #### 3. AdMob Test Banner Visible in tr/05-recipe-detail.png
> **Deviation type:** Informational note (not a deviation or defect)
> **Observation:** Turkish recipe detail screenshot shows a Google AdMob test banner at the top: "a test ad from go!". This is real free-tier app behavior in the simulator.
> **Impact:** None. Reviewers expect test ads in free-tier apps. Not a compliance issue.

**Why it was wrong:** The observation that the banner is "real free-tier app behavior" is correct, but the conclusion that it is "not a compliance issue" missed the actual problem. The banner was a visible signal that the app uses AdMob — yet Phase 27's PrivacyInfo.xcprivacy, POLICY-UPDATE.md, 27-01-SUMMARY.md, and notes.txt all failed to disclose AdMob. The compliance issue was not the banner itself; it was the missing disclosures around the SDK that produces the banner.

**Phase 27.1 correction:** Reclassify as **"Compliance gap closed in Phase 27.1."** The screenshot remains correct (test banners are expected in free-tier review builds). The four artifact updates in this phase close the documentation gap that the screenshot exposed.

**Authoritative reference going forward:** Read `27-VERIFICATION.md` Deviation #3 in conjunction with this section. The screenshot is fine; the missing disclosures it exposed are now resolved.

## Anti-Patterns Avoided

- ✅ **Did NOT modify Phase 27's verification report.** Corrections are documented in this report only. `git diff` confirms `.planning/phases/27-app-store-compliance/27-VERIFICATION.md` is byte-unchanged.
- ✅ **Did NOT modify the existing Mapbox Coarse Location entry.** A SECOND CoarseLocation dict was added for AdMob. Apple's manifest schema allows duplicate data types with different purposes; both entries map 1:1 to App Store Connect privacy form rows.
- ✅ **Did NOT add api.spoonacular.com to NSPrivacyTrackingDomains.** Spoonacular remains correctly classified as a data processor.
- ✅ **Did NOT add speculative tracking domains.** The 5 existing Google domains from Phase 20 are unchanged. CONTEXT.md required "conservative addition only" and research found no Google official source confirming additional domains for banner-only AdMob.
- ✅ **Did NOT add NSPrivacyAccessedAPITypes entries for Google SDKs.** Google Mobile Ads SDK 11+ and UMP SDK ship their own PrivacyInfo.xcprivacy bundles; Xcode aggregates them automatically. Adding redundant entries to the app manifest would risk validation conflicts.
- ✅ **Did NOT use the older NSPrivacyCollectedDataTypePurposeAdvertising constant.** All four new entries use `NSPrivacyCollectedDataTypePurposeThirdPartyAdvertising` (third-party ad networks per Apple's 2025-2026 docs) or `NSPrivacyCollectedDataTypePurposeAnalytics` (Other Diagnostic Data).
- ✅ **Did NOT touch AdMob runtime code.** AdClient.swift, ConsentClient.swift, ConsentReducer.swift, BannerAdView.swift, AdCardView.swift are byte-unchanged.
- ✅ **Did NOT translate POLICY-UPDATE.md to Turkish.** Localization is deferred per CONTEXT.md.
- ✅ **Did NOT edit fastlane app description files.** Description AdMob disclosure is deferred to Phase 28.
